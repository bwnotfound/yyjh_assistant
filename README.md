# 小地图坐标 OCR 字符模板

本目录存放 OCR 识别小地图坐标 `(x, y)` 用的字符模板。

## 文件命名约定

| 文件名            | 对应字符 |
| ----------------- | -------- |
| `0.png` ~ `9.png` | 数字 0–9 |
| `lparen.png`      | `(`      |
| `rparen.png`      | `)`      |
| `comma.png`       | `,`      |

字符 `(` `)` `,` 在某些 shell / 路径正则下需要转义，所以模板文件名一律用纯字母。

## 切割规范

来源图: 「ROI 截取工具」生成的 `tools/roi_captures/minimap_coord_roi_*.png`。
建议至少准备两张：一张包含个位数（如 `(2,3)`）、一张包含十位数（如 `(12,16)`），覆盖所有数字字符。

每个模板必须满足:

1. **紧贴 bbox**。沿字符笔画外缘切，不留任何 padding。padding 会引入背景纹理，自适应阈值后变成噪点干扰 NCC 相关度。
2. **包含完整笔画**。所有笔画端点都必须在 bbox 内，宁可多一像素也不能切掉笔画。
3. **每个字符宽度独立**。本游戏字体是比例字体——`1` 比其他数字明显窄，括号也比数字窄。**不要为了对齐补 padding 强行等宽**。
4. **保持原始分辨率**。不要把 ROI 放大后再切——OCR 运行时也是直接在原始截图尺寸上做匹配，模板和待识别图必须分辨率一致。
5. **保持原始色彩**。OCR 模块内部统一做灰度+自适应二值化，模板和 ROI 走同一管线，输入端必须一致（都是原始 RGB 截图）。

## 推荐工作流

1. 用「ROI 截取工具」截一张 `(12,16)` 的 ROI（也可以是任意包含 0、2、3、6、9 等含弧线数字 + `,` 的坐标）；再截一张 `(2,3)` 之类的低位数字 ROI 补完 `1`、`4`、`5`、`7`、`8`。
2. 用 Windows 「画图」/ Photoshop / Krita 等任意工具，把每个字符按上述规范切出，保存到本目录。
3. 启动 routine 时，runner 会自动从本目录加载模板。**任一字符模板缺失会打 warning 但不报错**，缺失字符的位置 OCR 会读不到，最终正则解析会失败 → 自然降级为「OCR 失败」单帧丢弃。

## 重新标定的触发条件

以下情况需要重新切模板:

- 游戏字体/字号被官方更新了
- 游戏分辨率变了（如从 1920×1080 切到 2560×1440）
- 截图管线变了（PrintWindow 模式或 DPI 缩放变化）

普通玩家场景下这些事件几乎不会发生，模板一次切好基本永久可用。

## 调试

OCR 跑不通时:

1. 在 `RoutineRunner` 启动 log 里看「OCR 闭环启用」是否打印；没打印说明模板加载失败或 ROI 未配置。
2. 在 routine 运行 log 里搜 `OCR 阶段1超时未变化` / `OCR 阶段2超时未稳定` / `OCR 到达错位`：
   - **阶段1超时**：通常是点击没生效，或 ROI 框错位置（一直读不到合法坐标）。
   - **阶段2超时**：识别在抖动（连续帧坐标读出来不一致），通常是模板不够准——某些字符模板偏小漏笔画。
   - **到达错位**：识别稳定但和 target 不等。检查 path 是否计算错误，或某个字符模板被错认为另一个（比如 `1` 被识别为 `(`）。
3. 把 `app.core.ocr` 的 logger 等级调到 DEBUG，可以看到每帧 OCR 的原始字符串输出，对比期望值能立刻定位是哪个字符在被误识别。
